# Interview Guide: incident-briefing-api

## 30秒サマリー

- 障害メモを保存し、AIで暫定原因・顧客影響・次アクションを構造化するAPI。
- 原文保持 + 要約別保存により、再実行と監査ができる。
- 運用現場を意識した設計で、実務理解を示せる。

## 3分説明の順番

1. 課題設定: 障害初動で共有の質と速度が安定しない。  
2. ドメイン設計: 原文と要約を分け、速報性と検証性を両立。  
3. エラーハンドリング: AI失敗時は 502、DB失敗時はロールバックで復旧しやすくした。  
4. 運用視点: 共有先（オンコール/サポート）を意識した出力項目を定義。  
5. 拡張余地: 非同期キュー、通知連携、インシデント分類の自動学習。

## 深掘り質問に対する答え方

- なぜ要約だけ保存しないか: 後で誤要約を検証できなくなるため。  
- なぜ運用向け構造化か: 障害対応は文章よりも定型項目のほうが判断が早いから。  
- 学生として何を意識したか: 「動く」だけでなく、障害時の再現性と説明責任を意識した。
